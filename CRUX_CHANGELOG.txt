Crux changelog
==============
Newest entries first. Each entry: commit name, local time, what changed since
the previous entry. Crux is the exam-readiness layer built on top of Anki
(formerly "TopGRE"): it teaches the problem -> type -> method routing path and
orders study by points at stake.


"Crux: project README + model description docs (handin deliverables)"  (Jul 5 2026, 1:35 PM)
- Added CRUX_README.md: exam stated up front, AGPL-3.0 + Anki credit, build/run
  for both apps, architecture overview, the Rust-change note with the list of
  upstream files touched and merge difficulty, and an honest status section.
- Added CRUX_MODELS.md: one section each for the Memory, Performance, and
  Readiness models (formulas, ranges), the projected-score method, and the
  give-up rule. These are required section-12 deliverables.

"Crux: projected overall GRE Math Subject score from topology"  (Jul 5 2026, 1:00 PM)
- Readiness now maps topology performance to a real-scale projected GRE Math
  Subject score (200 to 990), shown with a likely range and a confidence badge
  (low until topology coverage is high), on both the home dashboard and the
  readiness page. Withholds the number when there isn't enough evidence.
- Method (documented in code): projected = 700 + topology_readiness x 200,
  clamped to 200..990. High baseline is deliberate: topology is the cluster that
  separates top scorers, so strength elsewhere is assumed and stated.

"Crux: self-hosted Anki sync server for desktop <-> phone"  (Jul 5 2026, 12:52 PM)
- Added tools/crux_sync_server.ps1 and SYNC_SETUP.md to run Anki's own (shared
  Rust engine) sync server locally, so the desktop app and AnkiDroid sync the
  same collection with no AnkiWeb account. Verified: it binds 0.0.0.0:8080 and
  responds. Uses Anki's built-in offline-first sync and revlog-merge conflict
  rule (reuse the shared engine, do not reinvent sync). Doc includes the exact
  desktop/emulator/phone endpoints and the 7b offline + conflict test procedure.
- Chose Anki sync over Firebase deliberately: Firebase would mean rebuilding a
  parallel sync + merge layer, which fights the spec's conflict tests.

"Crux: fix crash when reopening the readiness/router dialogs"  (Jul 3 2026, 9:24 PM)
- Reopening the Router drill or Readiness dialog crashed with "wrapped C/C++
  object ... has been deleted". Cause: the dialogs never told Anki's
  DialogManager they had closed, so it kept a dangling reference and, on reopen,
  called windowState() on the deleted C++ object.
- Fix: call aqt.dialogs.markClosed("RouterDrill" / "Readiness") in each dialog's
  reject(), matching how Anki's built-in dialogs (About, Browser, stats, etc.)
  behave. Verified with an automated open, close, reopen self-test: no crash.
- This is the real cause of "worked once then broke": the first open worked, the
  second (after closing) hit the stale reference.

"Crux: fix 403 'Unexpected API access' on custom page fetches"  (Jul 3 2026, 9:12 PM)
- The router drill and readiness AI/heatmap/tuning panels fetch custom Python
  endpoints directly. Anki's media server rejects any POST whose Content-Type is
  not application/binary (a CSRF guard), returning 403 and popping "Unexpected
  API access". Our fetches sent application/json, so the drill showed "Could not
  load (Error: 403)" and the warning appeared.
- Fix: all six custom fetches now send Content-Type: application/binary (the body
  is still JSON text, which the handlers json.loads). Verified end to end: the
  click opens the dialog, the fetch reaches the handler, and the drill loads.

"Crux: fix crash when clicking home dashboard actions"  (Jul 3 2026, 9:00 PM)
- Clicking a heatmap tile (and the Cram/Router/Readiness home buttons) crashed
  Anki. Root cause: opening a webview-based dialog synchronously from inside the
  deck browser webview's own click callback re-enters QtWebEngine and crashes.
- Fix: dispatch every home dashboard action on the next event-loop tick
  (QTimer.singleShot(0, ...)) so the click callback returns first. Verified the
  router/readiness dialogs now open cleanly from the home.

"Crux: fix overlapping tiles in the coverage/danger heatmap"  (Jul 3 2026, 8:43 PM)
- The home heatmap tiles are clickable buttons, and Anki's global button rule
  (margin plus content-box sizing) combined with an explicit width:100% made each
  tile wider than its grid cell, so they overlapped. Fixed with box-sizing:
  border-box, width:auto, min-width:0, and margin:0, and preserved the danger
  tint on hover. Added the same box-sizing to the readiness heatmap for safety.

"Crux: focus a type in the router drill + AI proxy fix"  (Jul 3 2026, 5:10 PM)
- Router drill can now focus a single topology type (a Focus picker), and the
  home heatmap tiles are clickable: click a family to jump straight into a drill
  focused on it. This closes the triage loop: see the danger, drill exactly that.
- Fixed the AI integration so it actually calls the model:
  - crux_ai now also reads the documented OPENAI_BASE_URL / OPENAI_API_KEY (and
    TOPGRE_AI_MODEL), not only the CRUX_AI_* names, so an existing proxy setup
    works with no extra config.
  - Added a browser User-Agent to the request; Cloudflare was returning 403
    (error 1010, bot protection) for the default urllib agent. Verified live:
    the proxy now returns 200 and the coach/explain use real model output.

"Crux: honesty/tuning panel on the readiness page"  (Jul 3 2026, 2:20 AM)
- Added a "Honesty and tuning" panel: edit the give-up thresholds (minimum
  graded reviews, minimum exam attempts, minimum coverage) and the interleaving
  toggle, then save. This surfaces the honesty rule (Crux withholds a score until
  there is enough evidence) as something you can see and adjust.
- New handlers crux_settings (read) and crux_settings_save (write on the main
  thread) back the panel; values persist in the collection config and sync.

"Crux: reviewer answer accents + active-recall router drill"  (Jul 3 2026, 2:08 AM)
- Reviewer: the Again button now reads rose and the recommended answer (default
  ease) is an indigo-violet gradient, so the grade choices carry meaning at a
  glance instead of four identical buttons.
- Router drill is now genuinely two-stage: after you route the problem to a type,
  it asks you to recall the move before revealing it (active recall), then shows
  the move, follow-up, and optional explanation. Keyboard: Enter/Space/M reveals
  the move, then Enter/N advances.

"Crux: coverage/danger heatmap on the home dashboard"  (Jul 3 2026, 1:56 AM)
- The home screen (deck browser) now shows the coverage/danger heatmap directly,
  computed in Python during render (no network needed). You see the triage
  picture the moment you open the app: which topology families carry the most
  points at stake, and which have no cards yet.
- High-danger tiles switch to light text for contrast; empty tiles are dashed.

"Crux: coverage/danger heatmap + router keyboard play"  (Jul 3 2026, 1:45 AM)
- Readiness page now has a "Coverage and danger by type" heatmap: one tile per
  topology move family, indigo intensity scaled to points at stake, dashed tiles
  for families with no cards yet. Makes the triage picture legible at a glance.
- New read-only handler crux_type_map aggregates cards and danger per type.
- Router drill is now keyboard-playable: press 1-4 to pick a type, Enter or N for
  the next problem, E to explain. Option buttons show their number key.

"Crux: Router drill 'Explain why' + reviewer button polish"  (Jul 3 2026, 1:32 AM)
- Router drill reveal now has an "Explain why" button. It explains what signal in
  the problem points to that move family and why the move resolves it, teaching
  the routing itself. Uses AI when configured, otherwise a template from the
  card's own method text (works offline).
- New read-only handler crux_ai_explain in mediasrv.
- Reviewer answer buttons (Again/Hard/Good/Easy) are now pill-shaped with a
  subtle hover lift and indigo shadow, respecting reduced-motion.

"Crux: AI coach (triage plan) with deterministic fallback"  (Jul 3 2026, 1:20 AM)
- Added the Crux coach to the readiness page: press "Get plan" and it returns a
  concrete next-actions plan built from the triage engine (your most dangerous
  routing families, coverage gaps, attempt count). It is labeled "engine plan".
- When an AI proxy is configured (env CRUX_AI_PROXY_URL / CRUX_AI_PROXY_TOKEN,
  the existing token-gated Cloudflare OpenAI proxy), the plan is rewritten by the
  model and labeled "AI plan". The OpenAI key never touches the client.
- New qt/aqt/crux_ai.py helper: short timeout, fails soft to the deterministic
  plan, so the app never hangs and the feature works offline. AI is additive.
- New read-only handler crux_ai_coach in mediasrv aggregates danger by move type.

"Crux: Router drill (problem to type to method) + toolbar/overview polish"  (Jul 3 2026, 1:05 AM)
- New flagship feature: the Router drill. It shows a problem stem and asks which
  problem type (move family) it calls for, then reveals the move that solves it.
  This trains the problem -> type -> method decision path, which is the spiky POV
  (route the problem, do not just recall facts).
  - Frontend: ts/routes/router (SvelteKit page) in the Corporate Trust style,
    with a progress bar, four-option type picker, correct/misroute feedback,
    move + follow-up reveal, and an end-of-session summary that lists the types
    you misrouted most.
  - Data: a read-only Python handler (crux_routing_drill in mediasrv) builds the
    drill items from the decision-chain notes already in the collection.
  - Scoring: each routing choice feeds the Performance model through
    RecordExamAttempt, so drilling routing also updates readiness and danger.
  - Wiring: RouterDrill Qt dialog, Tools menu item (Ctrl+Shift+D), and a
    "Router drill" button on the Crux home dashboard.
- Toolbar: indigo-tinted rounded hover on the top-bar items for cohesion.
- Overview (deck study screen): the counts + Study now panel is now an elevated
  card with a larger heading.
